= メモ本文エディタ（CodeMirror + AsciiDoc）ロードマップ

このドキュメントは、メモ本文編集を CodeMirror 6 で置き換え、段階的に AsciiDoc 向けの編集体験（ハイライト・補完・WYSIWYG ライト）を足していく際の**作成指針**をまとめたものです。

参考にした設計思想は、Markdown 向け WYSIWYG エディタの解説記事（Decoration + WidgetType で見た目だけプレビューに寄せ、カーソル位置では raw に戻す）です。

* link:https://qiita.com/engchina/items/5246eacbd29daeee7dc2[Qiita: No.1 Markdown Editor の WYSIWYG（engchina）]

kbmemo では **Stimulus**（`MemoBodyEditorController`）+ **Vite** で同様の契約を AsciiDoc に横展開します。CodeMirror 6 自体はフレームワーク非依存のため、React 記事の構成をそのまま流用できます。

== 方針の固定（すべてのフェーズ共通）

[cols="1,3"]
|===
|項目 |内容

|正（ソース・オブ・トゥルース）
|DB / フォームに渡す **AsciiDoc のプレーン文字列**。サーバー側の変換・表示の正は引き続き **Asciidoctor**（`MemoWikiLinks` 等）。

|CodeMirror の役割
|編集体験のみ。ハイライト・補完・WYSIWYG はすべて CM 側の「見せ方」。

|WYSIWYG の型
|ProseMirror 的な「別木構造」にしない。記事どおり **Decoration + Widget + フォーカスで raw に戻す**。

|カーソル契約
|ブロック／インラインからカーソルが**離れている**ときだけ装飾・マーカー非表示。カーソルを**入れたら生ソース**に戻す（ユーザーがソースを信頼できること）。

|デバウンス・保存
|既存の `memo-draft`（Stimulus）との **単一ソース同期**。CM と textarea（および PATCH ペイロード）の二重状態を増やさない。ドラフトは統合 PATCH・直列キュー方針を維持する。

|Turbo
|Stream / partial 差し替え時は `disconnect` で CM を破棄し、textarea 経由で再接続する。
|===

=== AsciiDoc とパーサ戦略

AsciiDoc は Markdown ほど公式な CodeMirror 6 用文法（Lezer 等）が揃っていない。フェーズ進行では次を前提とする。

* 当面は **StreamLanguage / 正規ベース / 最小 HighlightStyle** を段階的に厚くする（`codemirror-asciidoc` 等の利用も可）。
* 将来的に **Lezer grammar / Tree-sitter** を検討する場合は工数を別途見積もる。
* CM 側は**軽量パーズ**、厳密さが必要な確認はサーバー（Asciidoctor）に寄せる**二段**もあり得る。

== フェーズ一覧

各フェーズ末尾の **ゲート** を満たしてから次へ進む。

=== Phase 1 — CodeMirror で textarea を置換（機能パリティ）

* Vite に `@codemirror/state`, `@codemirror/view`, `codemirror`（`basicSetup`）など最小セットを追加。
* `MemoBodyEditorController` で textarea を送信専用（非表示）にし、CM をマウント。
* `EditorView.updateListener` で `textarea.value` と `memo-draft` が読む入力を一致させる。
* Turbo で partial が差し替わったら `disconnect` で破棄・再接続。
* アクセシビリティ: `aria-labelledby` / ラベル連携、エラー時フォーカス。

*ゲート:* 既存ドラフト PATCH・コミットの挙動が導入前と同じ。

*現状（2026-05）:* 実装済み。

=== Phase 2 — AsciiDoc 向け「軽い」言語サポート

* `codemirror-asciidoc`（StreamLanguage）または自前ルールでシンタックス相当を提供。
* 優先順位例: 見出し（`=`）→ リスト → コメント → 強調など。
* Wiki リンク `[[…]]` を **HighlightStyle / Decoration** で見やすく（未解決リンク用 Decoration とも相性がよい）。
* 将来の WYSIWYG と重なっても競合しないよう、Decoration の **レイヤ・マーク prefix** を設計で決めておく。

*ゲート:* 長文でのパフォーマンス、Decoration レイヤ方針の文書化。

*現状（2026-05）:* ハイライト + 完成済み `[[…]]` のマーク装飾まで実装済み。レイヤ prefix の明示設計は Phase 4 前に見直す。

=== Phase 3 — Wiki リンク入力支援

* `@codemirror/autocomplete`: `[[` でタイトル検索 API にデバウンスし候補表示。
* 確定時は `insertCompletionText` に任せず、**明示トランザクション**で `[[full_path/slug]]` 等を挿入（`closeBrackets` による `]]` 自動挿入との二重を避ける）。
* 候補は `policy_scope` と同義の JSON API（`MemoWikiCompletions` / `GET /memos/wiki_completions`）。
* 候補形式: タイトル、同一ディレクトリの slug、フルパス（`MemoWikiLinks` と整合）。

*ゲート:* 線形テキスト上で壊れた `[[` が減り、ユーザーがソースを理解しやすい。

*現状（2026-05）:* 実装済み。候補はタイトル表示・確定時スラッグ挿入。既存 `[[…]]` 内の再選択はリンク全体を置換。

=== Phase 4 —「WYSIWYG ライト」モード（記事の核心）

Qiita 記事の「離れたらプレビュー風／入ったら raw」を AsciiDoc 用に**狭い構文サブセット**から実装する。AsciiDoc 全体を一度に WYSIWYG にしない。

* *ブロック:* 見出し行のみマーカー（`=`）非表示 + タイポグラフィ装飾（Markdown の `#` と同型だが AsciiDoc は `=`）。
* *インライン:* `*bold*` / `_italic_` 等、ごく限定した構文から開始。
* *WidgetType:* タスクリストや簡単な admonition など、製品で許す AsciiDoc 構文に相当するものがあれば同様に検討。
* UX 契約（ソースは信頼できる・カーソル位置で見え方が変わる）をテストまたはこのドキュメントで維持。

*ゲート:* 上記契約を満たすデモ可能な構文範囲が決まっていること。

*現状（2026-05）:* `memo_body_editor/wysiwyg_lite.js` で初期サブセットを実装済み。

* 見出し: 行頭 `=`〜`======`（1〜6 段）。カーソルが行外のとき `=` マーカーを非表示し、見出し用タイポグラフィを適用。
* インライン: `*bold*` / `_italic_` / `` `text` `` / `+text+`（二重 `` / `++` も可。カーソルが範囲内のときはマーカー表示）。
* 除外: フェンスコードブロック（` ``` ` 行）、リンク装飾範囲内のインライン強調。
* `EditorView.atomicRanges` で非表示マーカー上のカーソル挙動を安定化。
* Wiki / AsciiDoc リンク: `wiki_link_wysiwyg.js`（`[[…]]`、`link:`、`<<xref>>`、URL）。スラッグ解決時はタイトル表示、解決済みはクリックで `/memos/:id` へ。

*Phase 4 で意図的に残していたもの:* リスト行 → **Phase 5a**、admonition・ブロック Widget → **5b 以降**（下記）。

=== Phase 5 — 複合ブロック（コスト・難易度が跳ねるゾーン）

Phase 4 までの Decoration ベース WYSIWYG を、**ブロック単位**に広げるフェーズ。
いきなり Widget やネストエディタに入らず、**5a → 5b → 5c** と段階を分ける（難易度の跳ね上がりを抑える）。

共通契約（Phase 4 と同じ）:

* 正は DB の AsciiDoc プレーン文字列。CM は見せ方のみ。
* **フォーカスなし**、または**カーソルがブロック外** → プレビュー風。
* **カーソルがブロック内** → 生 AsciiDoc。
* 厳密な構文検証は Asciidoctor（表示・コミット後）に寄せ、CM は軽量パース。

==== 着手前に固定する意思決定

[cols="2,3"]
|===
|論点 |メモ

|許す AsciiDoc サブセット
|リスト・admonition・表・コードブロック・図のうち、**製品で正式サポートする範囲**を列挙する（ロードマップ各サブフェーズに反映）。

|テーブル
|当面は **ソース編集のみ** とするか、将来セル Widget を目標にするか。

|Mermaid / 数式
|本文に使う予定があるか。なければ **表示は Asciidoctor プレビューのみ**（編集 CM では raw）でよい。

|Widget の導入基準
|Decoration で足りるもの（リスト行・admonition 見出し）は Widget にしない。折りたたみ・ネスト CM が必要なときだけ Widget。
|===

==== Phase 5a — リスト行 WYSIWYG ライト（推奨: 最初の着手）

*対象（案）:* 箇条書き・番号付きの行頭マーカー。例: `*` / `-` / `.` / `1.` など（AsciiDoc の list 行。入れ子は後続でも可）。
*方式:* `wysiwyg_lite.js` の延長。行単位 `Decoration.line` + マーカー部分の `Decoration.replace`。
*カーソル:* その行に入ったら raw（Phase 4 見出しと同型）。
*ゲート:* フェンス内・リンク範囲と競合しない。長文（数千行）で装飾再計算が実用速度。

*現状（2026-05）:* 実装済み（`wysiwyg_lite.js`）。`*` / `-` / `.` / `1.` 等の行頭リスト。マーカー非表示・段階インデント・カーソル行は raw。

==== Phase 5b — Admonition（NOTE / TIP / WARNING 等）

*対象（案）:* 行頭 `NOTE:`、`TIP:`、`IMPORTANT:`、`WARNING:`、`CAUTION:` と続く本文ブロック。
*方式（第1段）:* ラベル行のマーカー非表示 + ブロック用 line class（左ボーダー・背景）。**Widget は使わない**。
*方式（第2段・任意）:* 複数行ブロック全体を折りたたみ可能な `WidgetType`。
*ゲート:* ブロック境界のパース（空行・ネストリストとの境）を文書化。

*現状:* 未着手。

==== Phase 5c — フェンスコード / listing

*対象:* ` ``` ` フェンス、AsciiDoc listing（`----` / `....` 等はサブセット決定後）。
*方式（段階的）:*
** 第1段: ブロック背景・等幅・言語ラベル表示（Decoration / line mark）。編集は全文エディタのまま。
** 第2段: 折りたたみ（`foldGutter` または ViewPlugin）。
** 第3段: ブロック内ネスト CM（別パネルハイライト）— コスト大、必要時のみ。
*既存:* Phase 4 でフェンス内はインライン WYSIWYG をスキップ済み。
*ゲート:* lazy 読み込み（言語ハイライト用 grammar はブロック表示時のみ）。巨大メモでスクロール・再描画を計測。

*現状:* 未着手。

==== Phase 5d — テーブル

*対象:* 製品で許す表記のみ（例: pipe 表 `|===` のみ、など要決定）。
*方式:* セル単位 `WidgetType` は **最重**。ソース編集を維持しつつ、ヘッダー行だけ WYSIWYG する等の縮小案も可。
*ゲート:* 表記バリエーション一覧 + 「編集は常にソース」か「セル UI あり」かの製品判断。

*現状:* 未着手。Phase 5 の後半または別マイルストーン推奨。

==== Phase 5e — Mermaid / 数式・その他埋め込み

*方針（推奨）:* 編集 CM では **ソースのまま**（ハイライトのみ）。レンダリングはメモ表示・プレビュー（Asciidoctor）に一本化。
*代替:* 「プレビュー専用ペイン」をエディタ横に置く場合は、二重実行・同期コストを別途見積もる。
*ゲート:* 本文で実際に使う記法が決まっていること。

*現状:* 未着手。需要がなければスキップ可。

==== Phase 5 全体のゲート

* 複雑 Widget は **lazy 読み込み**（初期表示・スクロール_into_view 時）。
* **巨大メモ**（例: 10k 行級）での装飾ビルド時間・メモリを一度計測し、上限または打ち切り方針を決める。
* 各サブフェーズ（5a〜）ごとに PR を分け、レビュー可能な単位を保つ。

=== Phase 6 — Turbo・マルチタブ・将来拡張

* Turbo Stream で本文 partial が差し替わるときの状態マージ（選択範囲・undo 履歴）戦略。
* 「ソース / WYSIWYG-lite」トグルがあるなら `Compartment` で拡張パックをオンオフ。
* E2E（Playwright 等）は任意。補完と装飾の結合ポイントだけ自動テストすると安全。

== リスクと意思決定（早めに固定）

[cols="2,3"]
|===
|論点 |推奨

|AsciiDoc 全体を一度に WYSIWYG にするか
|**しない**。Phase 4 は構文サブセットを決めて増やす。

|CM 側パーサの厳密さ
|編集用は軽量でよい。公開プレビュー・コミット後表示は Asciidoctor を正とする。

|React 記事との差
|kbmemo は Stimulus。CM API はそのまま利用可能。

|DnD 等からのドラフト
|サイドバー更新が必要なら `Accept: text/vnd.turbo-stream.html` を `memo-draft` と揃える（任意・未着手の場合あり）。
|===

== 推奨マイルストーン順（まとめ）

. CM で textarea と同等保存（Phase 1）
. 軽ハイライト + Wiki Completion（Phase 2–3）
. Decoration ベースの WYSIWYG-lite（見出し → インライン → リンク）（Phase 4）
. ブロック拡張: リスト（5a）→ admonition（5b）→ コード（5c）→ 表（5d）→ 図（5e）（Phase 5）
. Turbo / マルチタブ / Compartment 切替（Phase 6）

記事が示す設計哲学（**ソース不変・カーソル駆動の raw 復帰**）を保ったまま、AsciiDoc 向けに段階的に寄せる。

== Wiki リンク用スラッグ（全体一意）

* DB の `memos.slug` は `{stem}-{memo_id}` 形式（例: `first-memo-42`）。**アプリ全体で一意**。
* Wiki リンクの推奨表記: `[[first-memo-42]]`（ディレクトリ移動の影響を受けない）。
* レガシー: `[[stem]]`（stem が scope 内で一意のとき）、`[[full_path/slug-{id}]]`。
* Git ファイル名は `{global_slug}.adoc`（`MemoRepository`）。旧 `{slug}-{id}.adoc` から移行済み。
